# Portable Test Evidence Framework

Use this framework to derive a project-native test strategy without importing project-specific commands.

## Test Map

Before changing tests or choosing checks, map the project-native commands:

| Field | Meaning |
|---|---|
| Command or evidence source | Exact local command, CI job, fixture, trace, script, or manual evidence source. |
| Scope | Unit, property, integration, e2e, docs, install, release, platform, or other local category. |
| Prerequisites | Toolchain, service, network, hardware, privileges, environment variable, fixture, or generated artifact. |
| Default status | Runs by default, opt-in, CI-only, release-only, privileged-only, or manual. |
| Skip behavior | Can skip, must hard-fail in CI, or cannot determine skip reliably. |
| Diagnostics | Logs, failing seed, minimized corpus case, backtrace, trace, screenshot, artifact, or command replay. |

The map is not a universal gate list. It is a way to choose evidence proportionate to the current risk.

## Layer Map

| Layer | Purpose | Typical Evidence | Default Treatment |
|---|---|---|---|
| L0 fast checks | Pure logic, parser, config, small units, smoke commands | unit tests, format/lint, parser validation, CLI smoke | run by default when cheap |
| L1 property or corpus | Broad input space and known examples | generated cases, fixtures, golden corpus, metamorphic relations | run when behavior has many states |
| L2 integration | Multiple modules or local services without high external risk | local integration tests, simulated devices, test containers | run when boundaries changed |
| L3 environment/e2e | Real user workflow or external system touch | e2e tests, privileged hardware/kernel/browser/network path | run only when safe and proportional |
| L4 adversarial | Races, safety, fuzz, sanitizer, concurrency, performance edges | sanitizer, stress, fuzz, timeout, fault injection | run when the change affects those risks |
| L5 independent oracle | Behavior checked against a separate model or source | reference implementation, formal oracle, independently generated vectors | use for high-risk state machines or protocols |

The project may name layers differently. Preserve local names when they exist and map them to this risk model in the report.

## Regression Proof

A useful regression test has three properties:

1. It targets the observed failure mode, not a neighboring happy path.
2. It fails on the old behavior, baseline, fixture, trace, or independent oracle comparison.
3. It passes after the candidate fix with no unrelated source changes.

If the test cannot be copied directly to a baseline because it depends on new symbols, prove the old behavior with an equivalent public-interface test, fixture, trace replay, property, or oracle comparison.

## Property And Metamorphic Tests

Use these when fixed examples are too narrow:

- property test: the same invariant should hold for many generated inputs;
- metamorphic test: related inputs should produce predictably related outputs;
- negative corpus: known invalid inputs should fail safely;
- regression corpus: minimized reproductions of known bugs should remain fixed;
- shrinker: failing generated inputs should reduce to a minimal reproducible case.

Do not hide a property failure by weakening the property. If the property is wrong, record why and replace it with a sharper one.

## Fixtures And Golden Files

Prefer canonical fixtures that come from the same source a user or system would exercise: real config files, public examples, captured traces, sample payloads, or documented protocol cases.

If a golden file is generated by the same implementation being tested, it may be tautological. Strengthen it with at least one independent signal:

- human-readable annotated source;
- structural invariants;
- external or reference implementation comparison;
- round-trip or negative tests;
- reproducible fixture rebuild command;
- reviewable diff of the generated artifact.

Binary fixtures should have a source, rebuild path, or provenance note when practical.

## Skip And Hard-Fail Policy

Local development may skip checks whose prerequisites are unavailable. CI, release, or other high-confidence contexts should have a way to turn important skips into failures.

When a check can skip, report:

- what prerequisite was missing;
- whether skip was expected in the current environment;
- whether a stricter mode exists;
- what risk remains while skipped.

Do not treat "all checks passed" as meaningful when important checks were silently skipped.

## Test Discovery Canaries

When the language or framework requires manual test registration, add or look for a meta-test, canary, or deterministic discovery check. Its purpose is to catch the case where a test file exists but is not actually run.

If no discovery canary exists and tests are manually wired, report that as a test framework gap.

## Environment-Touching Checks

Some checks require privileges, hardware, network access, slow services, or destructive state. Treat them as evidence sources with explicit safety boundaries:

- state the prerequisite;
- state whether the current run has it;
- run only when safe and in scope;
- if skipped, explain the confidence gap and next safe way to run it.

## Failure Diagnostics

For slow, flaky, sanitizer, race, fuzz, or environment-touching checks, record the diagnostic artifact the next agent should preserve:

- exact replay command;
- seed or minimized case;
- logs and stderr;
- generated corpus entry;
- backtrace or crash dump;
- environment and toolchain version;
- captured external state such as device, browser, kernel, service, or network condition.

## Proportionality

Choose tests by risk:

- config/doc-only change: parser, link, fixture, or docs checks may be enough;
- pure function change: nearby unit/property checks first;
- state machine/protocol change: regression, property/metamorphic, and oracle/corpus checks;
- concurrency/resource lifecycle change: sanitizer/stress/leak/race checks;
- user workflow change: e2e or representative smoke path;
- privileged/external path: simulated check first, real environment check only when safe.

## Evidence Language

Say:

- "checked in this scope";
- "regression coverage proven against baseline";
- "skipped because <reason>, residual risk is <risk>";
- "blocked because the required evidence source is unavailable."

Do not say:

- "approved";
- "guaranteed";
- "fully tested";
- "safe to merge" unless the human explicitly owns that release/merge decision.
